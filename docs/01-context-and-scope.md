# Ограниченный Контекст И Границы

## Название контекста

**Authentication and Identity Context**

## Назначение контекста

Контекст отвечает за регистрацию, аутентификацию и выпуск токенов доступа.
Он инкапсулирует правила безопасности учетных записей, управление ролями и жизненным циклом сессий.

## Ответственность

Контекст обязан:
1. Создавать учетные записи и проверять уникальность идентификаторов.
2. Подтверждать учетные данные и выдавать токены.
3. Управлять refresh-сессиями и отзывом токенов.
4. Предоставлять публичные ключи (JWKS) для валидации JWT.
5. Управлять ролями и статусами пользователей.
6. Хранить опциональную привязку аккаунта к организации через `org_id` (B2B-ready).

## Структура агрегата

```shell
UserAccount (Aggregate Root)
└── Credential
└── Session (RefreshToken)
└── Role (set)
```

## Внешние зависимости

Контекст зависит от:
- **Key Store**: хранение приватных ключей подписи JWT
- **Persistence**: база учетных записей и сессий
- **Time Provider**: проверка сроков действия токенов

Контекст не зависит от:
- доменных сервисов (children, billing и т.д.)
- UI или клиентских приложений
- transport-слоя (HTTP/GRPC)

## Точки интеграции

- **Входящие**:
  - регистрация
  - логин
  - refresh
  - logout
  - управление ролями

- **Исходящие**:
  - JWKS endpoint
  - события о создании/блокировке аккаунтов (опционально)
  - JWT claim `org_id` (опционально, если задан у аккаунта)

## Явные границы

В контексте запрещено:
- хранить доменные данные, не относящиеся к auth
- изменять бизнес-инварианты других сервисов
- выдавать токены без политики и проверки статуса
- включать B2B-логику доступа по организации без отдельного решения (сейчас только сохранение/проброс `org_id`)
