# Видение Архитектуры Auth Service

## Назначение

Auth Service обеспечивает регистрацию, аутентификацию и выдачу JWT для клиентов (`web`, `mobile`, `admin`).
Сервис централизует управление учетными записями, ролями и сессиями, не смешивая доменную логику других сервисов.

Цели:
- единая точка входа для регистрации и логина
- безопасная выдача и отзыв токенов
- управление ролями и статусом учетных записей
- совместимость с доменными сервисами через JWT/JWKS

## Бизнес-ценность

Сервис дает:
- единый механизм авторизации для 10+ сервисов
- изоляцию критичной security-логики
- предсказуемую и проверяемую модель прав доступа
- снижение дублирования auth-логики

## Границы

### В зоне ответственности
- регистрация пользователя
- логин и выдача access/refresh токенов
- обновление access токена через refresh
- отзыв токенов (logout)
- управление ролями и статусом учетных записей
- предоставление JWKS для проверки токенов

### Вне зоны ответственности
- доменная логика других сервисов
- профили пользователей (если не требуется для auth)
- платежи, подписки, бизнес-данные
- UI и клиентские приложения

## Ключевые понятия

- **UserAccount** — учетная запись пользователя (Aggregate Root)
- **Credential** — данные для входа (password, oauth)
- **Role** — роль пользователя (`admin`, `user`)
- **Session / RefreshToken** — сессия и обновление токенов
- **JWKS** — публичные ключи для проверки JWT

## Принципы

- **Security First** — минимизация surface area, строгие инварианты
- **Single Source of Truth** — единое место для регистрации и токенов
- **Clean Architecture** — домен независим от инфраструктуры
- **Explicit Policies** — доступ через политики, не через ad-hoc проверки
- **Auditability** — критичные операции логируются
